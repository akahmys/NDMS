# Document Management System (DMS) 詳細仕様書

## 1. システム概要
DMSは、ローカルPC内のフォルダとシームレスに連携し、ドキュメント（主にPDFファイル）のメタデータ管理、プレビュー表示、ソート、分類、および最終的なPDF成果物統合を直感的に行うためのフロントエンド完結型Webアプリケーションです。

### 1.1. システム要件
* **実行環境**: モダンブラウザ（Google Chrome, Microsoft Edge など）
  * ※ローカルファイルシステムに直接アクセスするため、**File System Access API** の対応が必須です。
* **デプロイメント**: 単一ファイル構成 (`index.html`)。サーバーレスでローカルの `file://` プロトコルから直接実行可能。

---

## 2. アーキテクチャとテクノロジースタック
* **言語**: HTML5, Vanilla CSS, Vanilla JavaScript (ES6+ クラスベース)
* **UIフレームワーク / スタイリング**: Tailwind CSS (CDN版), CSSカスタムプロパティ（CSS変数）を用いた中心的なテーマ管理
  * **デザインシステム**: `tailwind.config` を拡張し、プロジェクト固有のカラー（`ink`, `paper`, `accent`）を定義。ハードコードされたスタイルを配し、セマンティックな CSS クラスによる一元管理を採用。
* **アイコン**: Lucide Icons (CDN版)
* **外部ライブラリ (CDN/UMD)**:
  * **SortableJS** (`1.15.0`): ドラッグ＆ドロップによる並び替え・要素移動
  * **pdf-lib**: PDFファイルの結合、テキスト描画（目次作成）
  * **pdf.js** (`3.11.174`): PDFファイルの1ページ目の解析・サムネイル画像生成用のレンダリング

### 2.1. データ管理 (Source of Truth)
* 物理ファイルのファイル名には連番などの管理番号を含めず、番号付与と管理はすべて `project.json` 内で行われます。
* **書類番号 (`number`)**: システムによる自動採番は行わず、ユーザーが右ペインから手動で入力・管理します。

### 2.2. 文字コードと正規化
* OS（macOSのNFD、WindowsのNFC）間での濁点・半濁点を含むファイル名の不整合を防ぐため、すべてのファイル名・パス文字列に対して **Unicode NFC正規化 (`.normalize('NFC')`)** を適用しています。

---

## 3. 主要モジュールとクラス設計
全体は役割ごとにクラス分割され、`App` クラスが全体を統括しています。

### 3.1. `Utils` (ユーティリティ)
* **UUID生成**: `crypto.randomUUID` またはフォールバックの乱数生成。
* **NFC正規化**: `Utils.norm(str)` による文字化け・パス不整合の防止。
* **パスサニタイズ**: Windows禁則文字のアンダースコア置換。

### 3.2. `FileSystemService`
File System Access API の薄いラッパー。すべてのI/O操作を統括します。
* `selectDirectory()`: ユーザーにフォルダを選択させる。
* `readFileAsText()`, `writeFile()`: `project.json` などのテキスト/JSONの読み書き。
* `moveAndRenameFile()`: ファイルのカテゴリ間移動とリネーム。
* `renameDirectory()`: フォルダ（カテゴリ）のリネームと移動。
* **フォールバック機構**: モダンAPI `move()` が使えないブラウザ（macOS版Chromeなど）でのフォルダ移動・リネーム失敗を回避するため、再帰的コピーと移動元削除によるフォールバック処理を実装しています。
* `readFileAsUrl()`: 画像やiframeで表示するための `Blob / Object URL` を生成。

### 3.3. `ProjectService`
メタデータ (`project.json`) の管理と、物理ファイルシステムとの同期を担当。
* **初期化ロジック (`init`)**:
  1. **通常ロード**: `project.json` が存在する場合。
  2. **ディスカバリーモード**: `project.json` は無いがサブフォルダが存在する場合、それらを「カテゴリ」として自動登録。
  3. **自動セットアップモード**: 空フォルダの場合、組み込みのテンプレート（例: 荏原標準 HES-0010-02）を展開し、標準フォルダ群を物理生成。
* **物理同期 (`syncFilePositions`)**:
  アプリ起動時や更新時に、実ファイルとJSONの差分を解消する。
  * 移動されたファイルの追従更新。
  * 未登録ファイルの自動登録検出。
  * 紛失ファイルのメタデータからの自動登録解除（削除）。

### 3.4. `PdfService`
PDFの加工を行うサービス。
* **成果物生成 (`mergeProjectPdfs`)**: `project.json` に記述された順序（カテゴリ順 → 書類順）に従って全PDFを1ファイルに結合。
* **目次自動生成**: 結合PDFの先頭に、文書番号・タイトル・開始ページ番号を記載した目次ページを挿入。

### 3.5. `App` (メインアプリケーション)
UI全体の描画、イベントハンドリング、ドラッグ＆ドロップ制御を一手に担います。

## 4. ユーザーインターフェース (UI)
画面は3ペイン（左・中央・右）のプレミアムな「Ink and Paper」デザインで構成されています。
すべての UI 要素は CSS 変数（`--color-ink`, `--color-paper` 等）と共通コンポーネントクラス（`.btn-ink`, `.input-standard` 等）によって一貫性が保たれています。

### 4.1. 左ペイン: カテゴリツリー
* **未分類セクション**: ルート直下に存在する、どのカテゴリにも属していないPDFをアラート付きで表示。
* **カテゴリリスト**: フォルダとしてのカテゴリ一覧と、その配下の書類を表示。
* **ソート機能**: SortableJSにより、カテゴリ自体の順序入れ替え、およびファイル単位のドラッグ＆ドロップによる順序変更・カテゴリ間移動が可能。

### 4.2. 中央ペイン: サムネイルグリッド / PDFプレビュー
* **サムネイルグリッドビュー (`viewMode: 'grid'`)**:
  * カテゴリ選択時に表示される、PDFの1ページ目を描画した美しいカード型グリッド。
  * `pdf.js` ワーカーによる非同期レンダリング。キャッシュ制御により高速・省メモリ化。
  * 共通クラス `.thumb-card` による立体的なホバー・アクティブアニメーション。選択状態は `.selected` クラスで管理。
* **PDFプレビュービュー (`viewMode: 'preview'`)**:
  * アイテムのダブルクリック、または「PDFを全画面で表示」ボタンから切り替わる `iframe` を使ったライブプレビュー。
* **一括操作サマリー**:
  * 複数選択（ShiftキーまたはCtrl/Cmdキー）した場合、中央および右ペインが「複数選択時のサマリー画面」に切り替わり、一括ドラッグ移動などの操作ガイドを表示。

### 4.3. 右ペイン: 情報・プロパティペイン
* **リアルタイム自動保存機構**: 保存ボタンはなく、`change`イベントを検知してIME確定後やフォーカス離脱時に自動で変更を反映し、必要なファイルシステム操作（リネームなど）も即座に行われます。
* **案件全体設定（上部・常時表示）**: JSONの `config` に該当する情報（案件名、受注番号、WBS、顧客名、使用先など）の入出力を行います。
* **カスタム項目**: 最大3つまで、ラベルと値をユーザーが動的に追加・編集できるカスタム項目機能を備えています（`config.customFields` に配列として格納）。
* **書類詳細（下部・選択時表示）**: 選択中のPDFの表示名（書類名）、書類番号、作成日、所属カテゴリの表示・変更を行います。変更時に物理ファイル名も連動して追従リネームされます。

## 5. 主要なユーザーフロー

1. **初期起動**:
   「図書フォルダを選択」ボタンから対象フォルダを選択。システムが状態を判定し、UIを構築。
2. **ファイルの追加（外部ドロップ）**:
   OSからカテゴリへ直接ファイルをドラッグ＆ドロップすると、アプリがファイルを該当フォルダへ物理コピーし、同名衝突時は `(1)` などの連番を付与して自動登録。
3. **ドキュメントの整理**:
   サムネイルグリッドまたはツリー上でアイテムをドラッグし、順序変更や他カテゴリへの移動を行う。複数選択による一括移動も可能。
4. **テンプレート変更（フォルダ構成の移行）**:
   「フォルダ構成変更」機能により、異なる業務テンプレート（例：標準用から排水ポンプ車用など）へ切り替え可能。その際、既存ファイルを新しいフォルダのどこにマッピングするかをUI上で選択し、一括で物理移動・メタデータ再編を実行する移行ウィザードが実装されている。
5. **最終出力**:
   整理が完了したらヘッダーの「成果物生成」を実行し、目次付きの結合PDFを出力・ダウンロードする。

## 6. 今後の保守・拡張に関する制約事項
本システムを改修する際は、以下のルール（`.gemini/rules.md` に基づく）を厳守すること。
1. **事前承認**: いかなる変更も必ず実装計画を提示し、ユーザーの承認を得ること。
2. **単一ファイルの維持**: すべてのコードは `index.html` に収める。外部アセットは引き続きCDNを利用すること。
3. **NFC正規化の徹底**: Mac/Windows間での互換性のために `Utils.norm()` によるファイルパス正規化を外さないこと。
4. **手動検証**: ブラウザ自動化を利用しないこと。検証は必ずユーザー自身（手動）に依頼すること。
